# 로컬 및 원격 P2P 파일 전송 기능 설계서 (P2P File Transfer Feature Design Spec)

* **작성일**: 2026-07-18
* **상태**: 승인됨 (Approved)
* **작성자**: Antigravity (AI Coding Assistant)

## 1. 개요 및 목표 (Overview & Goals)

본 기능은 사용자가 이 프로젝트의 웹 도구(PDF 병합, 변환, 증명사진 등)를 사용하여 작업한 결과물 파일이나 임의의 로컬 파일을 다른 기기(특히 모바일폰, 태블릿, 혹은 타 PC)로 간편하고 빠르게 전송할 수 있는 기능을 추가하는 것을 목표로 합니다.

### 핵심 원칙
1. **100% 로컬 브라우저 실행**: 파일의 원본 바이너리 데이터를 저장하거나 중개하는 백엔드 서버를 두지 않습니다.
2. **완벽한 개인정보 보호**: 파일 전송은 WebRTC 기술을 활용해 두 기기간 1:1 직접 연결(P2P)로만 흐르며, 파일 데이터는 암호화 전송됩니다.
3. **편의성 극대화**: 로컬 무선망(동일 Wi-Fi) 내에서는 자동 감지(Snapdrop 방식)를 지원하고, 타 네트워크(LTE 등) 환경에서는 카메라 QR 코드 스캔 1회만으로 즉시 다운로드되는 하이브리드 UX를 제공합니다.

## 2. 요구사항 및 사용자 경험 (Requirements & UX)

### 주요 기능
1. **독립형 파일 공유 도구 (`/share` 페이지)**:
   - 메인 화면에 "기기 간 전송" 도구가 추가됩니다.
   - 드래그 앤 드롭으로 모든 종류의 파일을 업로드하여 수신 대기 상태로 진입할 수 있습니다.
2. **기존 도구 완료 화면과의 통합 (`ShareModal`)**:
   - PDF 병합/분할, 워터마크, 회전, 변환, OCR, 증명사진 등 기존 모든 도구의 작업 완료 페이지에 **[기기 간 전송]** 버튼이 추가됩니다.
   - 버튼 클릭 시 완료된 결과물 파일을 물고 즉시 공유 대기 모달(`ShareModal`)이 팝업됩니다.
3. **하이브리드 P2P 공유 화면**:
   - **자동 발견 (안 A)**: 동일한 공인 IP(동일 Wi-Fi) 하에 공유 페이지를 띄워놓은 기기가 있으면 아바타 아이콘으로 탐색창에 자동 감지되어 보여집니다. 아이콘 클릭 시 파일 전송이 바로 시작됩니다.
   - **QR 코드 / 다이렉트 링크 공유 (안 B)**: 송신 측 화면에 고유 연결 정보(Peer ID)를 품은 QR 코드와 공유 URL이 실시간 표시됩니다. 수신 측(모바일)에서 이 QR 코드를 카메라로 한 번 스캔하면 브라우저가 열리며 자동으로 다운로드가 개시됩니다.

## 3. 아키텍처 및 상세 구현 설계 (Architecture & Technical Details)

### 3.1 파일 및 모듈 구성

* **독립 공유 페이지**: `frontend/src/app/share/page.tsx`
* **공유 모달 컴포넌트**: `frontend/src/components/ShareModal.tsx`
* **WebRTC 공통 로직 훅**: `frontend/src/hooks/useWebRTCShare.ts`
* **자동화 E2E 테스트**: `frontend/tests/share.spec.ts`

### 3.2 기술 스택 및 라이브러리
- **WebRTC 라이브러리**: `peerjs` npm 패키지를 사용합니다. WebRTC의 SDP Offer/Answer, ICE Candidate 교환 작업을 단순화하여 복잡한 시그널링 버그를 원천 차단하고 번들 사이즈를 최소화합니다.
- **공개 시그널링 클라우드**: PeerJS가 무상 제공하는 공개 시그널링 서버(`0.peerjs.com`)를 최초 중개 수단으로 사용합니다.

### 3.3 무서버 자동 매칭 매커니즘 (공인 IP 해싱)
동일 Wi-Fi 내의 자동 발견 기능을 무서버로 구현하기 위해 클라이언트 단에서 아래와 같이 동작합니다.
1. 기기 접속 시 `https://api.ipify.org?format=json` 등 공개 IP 조회 API를 호출하여 현재 단말의 공인 IP를 획득합니다.
2. 획득한 IP 문자열을 클라이언트 단에서 SHA-256 해시 처리하여 유일한 룸 키(`room-[IP_HASH]`)를 생성합니다.
3. PeerJS 시그널링을 통해 해당 가상 룸에 참여함으로써, 동일 공유기 아래에 있는 단말들만 서로의 연결용 Peer ID를 리스팅하고 교환할 수 있게 합니다.

### 3.4 대용량 전송 흐름 제어 (Backpressure)
WebRTC Data Channel의 대역폭 한계로 인한 브라우저 크래시나 전송 유실을 막기 위해 흐름 제어를 적용합니다.
1. 송신측은 파일을 **16KB(16,384바이트) 단위의 ArrayBuffer 청크**로 나눕니다.
2. 데이터 채널의 `bufferedAmount`를 감시합니다.
3. 버퍼가 차면 전송을 일시 정지하고, 채널의 `bufferedamountlow` 이벤트가 발생해 버퍼가 안전선 이하로 비워진 것을 확인한 뒤 다음 청크를 발송하는 백프레셔(Backpressure) 로직을 탑재합니다.
4. 수신측은 청크들을 메모리 상의 배열에 차곡차곡 쌓고, 메타데이터(전체 크기)와 대조해 수신 완료가 확인되면 `new Blob(chunks, { type: fileType })`을 생성하여 브라우저 API로 파일 다운로드를 트리거합니다.

## 4. 예외 처리 (Error Handling)

- **Symmetric NAT 연결 차단 및 타임아웃**:
  사내 프록시나 철저한 방화벽 내부망에서 WebRTC direct P2P 연결이 차단된 경우, 15초간의 시도 끝에 타임아웃을 트리거합니다. 화면에 경고 문구와 함께 *"동일한 Wi-Fi 공유기에 연결하거나 VPN을 종료해 보세요"*라는 우회 가이드를 제시합니다.
- **송신 브라우저 종료**:
  전송 과정 중 송신자가 탭이나 브라우저를 닫을 경우 수신 측에서 `close` 이벤트를 수신하여 다운로드가 중단되었음을 사용자에게 친절히 안내합니다.
- **메모리 한계 고지**:
  모바일 브라우저의 RAM 한계를 고려하여 수백 MB 이상의 파일 공유 시도 시 *"모바일 기기 전송 시 메모리 상황에 따라 중단될 수 있습니다"*라는 안내 툴팁을 제공합니다.

## 5. 검증 및 테스트 계획 (Verification & Testing)

### 자동화 E2E 테스트 (Playwright)
Playwright의 다중 브라우저 컨텍스트 실행 능력을 이용해 100% 로컬 환경에서 테스트를 구동합니다.
- **테스트 파일 위치**: `frontend/tests/share.spec.ts`
- **시나리오**:
  1. 가상의 `senderContext`에서 `/share` 페이지에 접근, 모의 파일(`dummy.pdf`)을 업로드합니다.
  2. 화면의 복사용 인풋 창에서 원격 접속용 공유 URL을 추출합니다.
  3. 가상의 `receiverContext`에서 추출된 공유 URL로 접근합니다.
  4. 송수신 브라우저 간의 시그널링 및 P2P WebRTC 데이터 채널 수립을 대기합니다.
  5. 송신 브라우저가 전송 버튼을 누르면, 수신 브라우저 단에서 `waitForEvent('download')`가 성공적으로 끝나는지 감지합니다.
  6. 다운로드된 파일의 크기와 무결성(MD5 해시 또는 바이트 매칭)이 원본과 정확히 일치하는지 비교 검증합니다.
